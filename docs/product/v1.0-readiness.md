# v1.0 Readiness Matrix

Granular companion to the [`v1.0 Criteria`](roadmap.md) section of the roadmap. The roadmap names 8 high-level criteria; this doc tracks the dozens of feature-level acceptance items, the open questions still to decide, and the manual workflow passes that should run before tagging v1.0.

> **History note.** This document was originally `beta-acceptance.md`, framed around a beta tag that never shipped. csaw has since released 11 minor versions in production use without a formal "beta" milestone. The matrix and gates framework were too useful to discard, so they're reframed here as v1.0 readiness gates. Status column reflects current state through v0.8.3 — see [`next-up.md`](../planning/next-up.md) for active tactical work.

## v1.0 Narrative

The v1.0 promise is:

> Repos remain the source of truth for project-owned context. csaw activates the right external AI workspace context for a repo, composes it from approved sources, projects it into the tools people use, and audits that the wrong context is absent — all reversibly, with no daemon, database, or hidden state.

This is the same narrow promise the product has held since v0.4.x. csaw is not a personal operating system for work modes, a SaaS control plane, a hard sandbox, or a replacement for git, `direnv`, Nix, `mise`, devcontainers, IDE settings, or task runners.

## Scope Rules

- A configured source is inventory. It must not imply active context.
- Active context is the mounted state in a project plus any source pins.
- Composition is explicit through profile activation (`csaw use`), advanced mount selection, priorities, and protected files.
- Client isolation is assurance: csaw can detect wrong local context and policy drift, but it does not sandbox an AI runtime.
- Context that belongs to exactly one project and can be safely committed should usually stay in that project repo.
- v1.0 docs must explain when not to use csaw as clearly as when to use it.

## Release Gates

| Gate | Definition | v1.0 Requirement |
|---|---|---|
| Works | The feature behaves correctly in normal and expected edge cases | Required for all public commands |
| Makes sense | The feature has a clear reason to exist against the repo-local objection | Required for all first-screen README and curriculum examples |
| Explains itself | Output and docs make active context, source provenance, and failure recovery obvious | Required for mount, audit, inspect, status, check, pin, and unmount |
| Protects data | Existing repo files are not overwritten, lost, or hidden without an explicit reversible path | Required for mount, adopt, force, restore, keep, unmount, fork, and promote |
| Is honest | Security and isolation language does not overclaim enforcement | Required across README, roadmap, curriculum, and policy docs |
| Is testable | Behavior has automated tests or a named manual workflow pass | Required before v1.0 tag |

## Acceptance Status Taxonomy

- **v1.0 must-have:** should be fixed or verified before tagging v1.0.
- **v1.0 decision:** choose whether v1.0 requires this, then either promote it to must-have or explicitly defer it to post-v1.0.
- **v1.0 caveat:** can ship if docs and command output set the right expectation.
- **Post-v1.0:** important, but not required for a polished v1.0.

## Recommended v1.0 Scope

v1.0 should include:

- install and release reliability across Homebrew/Scoop/PyPI/GitHub Releases
- source lifecycle (long-form + shorthand) for personal, team, and client registries
- explicit profile activation and profile-based composition
- safe mount, unmount, restore, force, keep, and skip-conflict flows
- kind-aware mounting for all 7 kinds (instructions, rules, agents, skills, MCP, hooks, ignore)
- tool projection into the 7 in-code tools (with codegen + CI consistency)
- priority/protected-file behavior with understandable provenance (layered detail is the open gap)
- project adoption from existing AI workspace files
- client isolation through required and blocked sources
- source URL/ref/pin audit checks (including Source.Ref vs per-project pin precedence)
- protected-file drift detection
- fork and promote contribution loops
- `status`/`check`/`inspect` output good enough to explain the active context (status now flags source-checkout dirty edits as of v0.8.2)
- `experimental/` convention with explicit `--include-experimental` override
- docs and curriculum that can be followed without guessing

## Open Questions Before v1.0

These four were the v0.4.x-era "must-fix decisions before tagging beta." Three are still open in v0.8.3, with refined framing:

1. **Layered provenance in `inspect`.** Show not only winning source but also losing candidates and whether priority or protection decided the result. Same gap also affects the "Priority composition" matrix row. **Should this block v1.0, or be a v1.x follow-up?**
2. **`diff` behavior.** Current `diff` semantics are weak for symlinked mounted files. **Fix the implementation, or rescope and document the limitation?**
3. **Client policy template.** Today `audit --init` writes a minimal template. Earlier plan was to add a `--template client` variant. **Required for v1.0 if client isolation is the wedge, or are exact docs enough?**
4. **`csaw context status/use/leave` vocabulary refactor.** *(Closed: deferred to post-v1.0 per refreshed roadmap. The current `csaw use <source>/<profile>` works; no user has reported confusion that would justify the breaking-change cost.)*

## Workflow Pass Plan

These eight passes are the manual verification gates before tagging v1.0. Status notes reflect the work tracked in [`exec-plans/active/v1.0-readiness-audit.md`](../exec-plans/active/v1.0-readiness-audit.md).

### Pass 1: Fresh User Path

Goal: verify a new user can install, initialize, mount, inspect, audit, and unmount without prior csaw knowledge.

Required checks:

- create a fresh project and fresh local registry
- run `init` (try both default and `--preset` variants)
- mount default context
- inspect mounted files and active sources
- run `status`, `check`, and `audit`
- unmount and restore
- repeat using only docs/curriculum instructions

### Pass 2: Existing Project Adoption

Goal: verify csaw can adopt a project with existing AI workspace files without losing project context.

Required checks:

- create project-owned `AGENTS.md`, tool rules, skills, and MCP-like files
- run `init --adopt`
- verify adopted files preserve project content
- verify starter files do not overwrite project content
- verify pre-existing registry files are preserved
- mount, unmount, and restore after adoption

### Pass 3: Composition And Client Isolation

Goal: verify the product story that configured sources are inventory and active context is explicit.

Required checks:

- configure personal, team, client-acme, and client-globex sources
- mount client-acme only
- verify client-globex is not mounted
- mount an explicit acme+personal profile
- verify priority and protected behavior
- run audit for correct client, wrong client, missing client, blocked client, and strict mode
- inspect source winners and any hidden composition ambiguity

### Pass 4: Conflict And Data Safety

Goal: prove local files are not lost and users can recover from mistakes.

Required checks:

- mount over existing files with default behavior
- test `--skip-conflicts`
- test `--force`
- test `--keep`
- test restore after unmount
- test partial unmount
- remove or edit mounted links and run `check`/repair flows
- verify `.git/info/exclude` and stash state cleanup

### Pass 5: Remote Source And Pinning

Goal: verify git-backed source behavior without relying on external services.

Required checks:

- use local bare remotes for team/client sources
- add, pull, push, fork, and promote files
- pin source to branch, tag, and SHA where supported
- verify Source.Ref (set via shorthand `#ref`) is honored as default
- verify per-project pin takes precedence over Source.Ref
- pin a branch that is already checked out in the source clone
- unpin while files are actively mounted
- audit required URL/ref/pin conditions

### Pass 6: Tool And Kind Projection

Goal: verify each public kind and supported tool path works.

Required checks:

- mount each kind independently (all 7)
- mount all kinds together
- verify instructions, rules, agents, skills, MCP, hooks, ignore land in expected tool directories
- verify Copilot suffix rewriting (`rules/x.md` → `.github/instructions/x.instructions.md`)
- verify kind filters do not accidentally mount omitted higher-risk kinds
- verify output counts actual mounted files and tool dirs

### Pass 7: Error And Malformed Input

Goal: make v1.0 failures feel deliberate instead of accidental.

Required checks:

- malformed `csaw.yml`
- unknown source
- unknown profile
- invalid kind
- bad pin ref
- malformed shorthand (`gh:` with no path; unknown prefix)
- missing worktree/source checkout
- noninteractive conflict
- policy schema mistakes
- JSON output for audit failures

### Pass 8: Cross-Platform And Release

Goal: verify v1.0 can be installed and run by real users.

Required checks:

- CI passes on Linux, macOS, and Windows
- `go test ./...`, `go vet ./...`, `go build ./...` pass locally before tag
- release workflow produces expected artifacts
- Homebrew, Scoop, PyPI, GitHub Release docs match the tag
- Windows link fallback behavior covered by CI or a named manual run
- **Windows in-the-wild testing** (the honest gap per refreshed roadmap) — at least one real user on Windows confirms install + mount + unmount before v1.0

## Architectural Decision Record

| Decision | Answer | Rationale |
|---|---|---|
| Should v1.0 be framed as "Personal Operating System For Work Modes"? | No | The idea is compelling but broader than current functionality. Research track only. |
| Should v1.0 lead with context-switching language? | Carefully | "AI workspace context activation" is accurate now. Heavier `context status/use/leave` vocabulary deferred per refreshed roadmap. |
| Should client isolation be the wedge? | Yes | Sharpest answer to "why not keep files in the repo?" because it involves privacy, provenance, pins, and absence checks across projects. |
| Do configured sources bleed across projects? | No | Configured sources are inventory. Only mounted sources/profiles are active. Docs and output reinforce this. |
| Is audit enforcement? | No | Audit is local assurance and policy drift detection. Hard enforcement would require runtime/OS/IDE/container controls outside current csaw. |
| Should csaw match cc-switch / APM / ECC feature lists? | No | Per the "user-driven over parity-driven" principle: features added when a real user reports friction, not because a competing tool ships them. |
| Should csaw ship a desktop GUI? | No (post-v1.0 at earliest) | CLI-first identity until clear CLI PMF. cc-switch's 81k stars solve a different (provider/key) problem. |

## Exit Criteria

v1.0 is ready when:

- every "v1.0 must-have" matrix row is verified or has an issue explicitly blocking the tag
- "v1.0 decision" rows are either promoted to must-have or explicitly deferred to post-v1.0 in this doc
- all four open questions above are resolved
- the workflow pass plan has been run end-to-end with notes captured in `exec-plans/active/v1.0-readiness-audit.md`
- README, curriculum, roadmap, audit docs, and this matrix tell the same story
- release validation passes on the commit being tagged
- **at least one real user on Windows** confirms install + mount + unmount (the in-the-wild gap)
